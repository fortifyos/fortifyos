# R-04 — Database Schema Drift / No Migrations

**Category:** Architecture / Reliability
**Severity:** High

## Description

The SQLite database had no migration system. Schema changes required manual intervention or destructive operations. There was no version tracking, no upgrade functions, and no rollback path. The `audit_log` table structure had evolved without a recorded change path, making it impossible to reliably upgrade a production database or understand what version a given database was running.

## Impact

As the schema evolved through development, existing databases could not be upgraded automatically. A production deployment with user data could not safely apply schema changes without migration logic. Rollback was entirely manual. There was no way to determine whether a running database was current, ahead, or behind the expected schema.

## Evidence

- `src/db/sovereignDB.js` — no `LATEST_SCHEMA_VERSION`
- No `upgradeToV3()`, `upgradeToV4()` functions
- No migration path for `audit_log` changes
- No `NOT NULL DEFAULT` enforcement on new columns

## Required Fix

Introduce explicit migration functions with version tracking. Each schema change gets a numbered upgrade function. The database must record its current version on upgrade and be capable of applying sequential upgrades from any lower version to current. Rollback documentation must exist even if automation is not implemented.

## Status

**RESOLVED** — `src/db/sovereignDB.js` now has `LATEST_SCHEMA_VERSION = 4`, explicit `upgradeToV3()` and `upgradeToV4()` functions with schema version tracking in `settings` table. Committed `f611feb8`. Rollback policy documented (manual, not automated).
