# R-05 — No Alerting or Observability Layer

**Category:** Reliability / Architecture
**Severity:** High

## Description

The system had no structured alerting mechanism. Operations that should trigger notifications — security failures, critical errors, state changes — had no way to emit alerts to operators or users. The `alerts/` directory did not exist. There was no `AlertEmitter`, no `AlertDispatcher`, and no adapter system for routing alerts to different destinations.

## Impact

Operators had no visibility into critical events. Security failures, database errors, and system anomalies went unreported. There was no way to subscribe to alert categories, no deduplication, and no way to route alerts to external systems (webhooks, notification services). The system could fail silently at any point with no operator awareness.

## Evidence

- `src/alerts/` directory did not exist
- No `AlertEmitter` class
- No `AlertDispatcher` with adapter routing
- No `ConsoleAdapter`, `BroadcastAdapter`, `NotificationAdapter`, or `WebhookAdapter`
- No dedup mechanism (per-tag cooldowns)
- No graceful degradation when adapters fail

## Required Fix

Implement an `AlertEmitter` with typed events and per-tag deduplication. Add an `AlertDispatcher` that routes alerts to the first available adapter. Implement at minimum a `ConsoleAdapter` and a `BroadcastAdapter`. All adapters must degrade gracefully when unavailable. No alert emission should crash the calling process.
