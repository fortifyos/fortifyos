# R-08 — Verifier Integrity Violation

**Category:** Governance / Verification
**Severity:** Critical

## Description

The verifier was mutable during execution, allowing test definitions to change while validating the same task. During TASK-FOS-002 validation, regex patterns were modified mid-run until tests passed. This broke the fundamental ALFRED verification guarantee: that a locked verifier produces deterministic results.

## Impact

False-positive validation — tests could be changed until they pass rather than code being correct. Complete breakdown of ALFRED verification guarantees. Inability to trust "all tests pass" as a meaningful statement. The entire ALFRED verification contract depends on immutable, locked verifiers.

## Evidence

- TASK-FOS-002 initial run: regex patterns modified during validation
- Test output discrepancy: first run showed 22/24, corrected run showed 24/24
- The correction required re-executing from a stable base with a locked verifier
- No RULE-VER-003 equivalent enforced at the time

## Required Fix

Lock verifier before task execution begins. Prohibit any mutation of verification logic during an active task. Treat verifier changes as separate, independently tracked tasks with their own execution contract. Enforce RULE-VER-003 equivalent: verifier is immutable once a task execution contract is declared.

- TASK-FOS-002 correction — Re-executed from stable base with locked verifier, 24/24 confirmed
