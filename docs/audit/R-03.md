# R-03 — Silent Security Failures

**Category:** Security / Reliability
**Severity:** Critical

## Description

Security-sensitive operations in the signing, passkeys, and crypto modules threw untyped, unstructured errors. `throw new Error(failReason)` with no audit trail. Silent `catch` blocks swallowed exceptions without logging. Error codes were not semantically meaningful — a signing failure looked identical to a timeout.

## Impact

Security failures were indistinguishable from operational errors. No audit log was written when signing or key operations failed. Silent catches meant the system could fail silently and return incorrect results, leaving the caller unaware that a security operation had failed. Forensic reconstruction was impossible without application logs.

## Evidence

- `src/security/vaultSigning.js` — bare `throw new Error()` throughout, no `secureLogRef`
- `src/security/passkeys.js` — bare `throw new Error()`, no structured error class
- `src/crypto/sovereignCrypto.js` — bare `throw new Error()`, no typed error hierarchy
- Silent `catch` blocks in all three modules
- No audit logging on security-critical paths

## Required Fix

Introduce typed error class hierarchy for all security operations. Each error class must carry a semantic code, a human-readable message, and a flag indicating whether the error should be audited. All `catch` blocks must either handle the error or re-throw with added context. Security failures must be logged via `secureLogRef()` before any return.
