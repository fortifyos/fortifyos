# R-02 — Unvalidated API Inputs

**Category:** Security
**Severity:** Critical

## Description

The agent API had no input validation layer. No checks on required fields, no enum enforcement for capability requests, no bounds on array inputs, and no rejection of unknown field names. The system accepted any payload shape without complaint.

## Impact

An unauthenticated or compromised caller could send arbitrary payloads. Without validation at the boundary, bad data propagated directly into signing operations, capability lookups, and database writes. No error code was returned for malformed requests — only silent failure or TypeError at unpredictable points.

## Evidence

- `src/agents/api.ts` — no `validateAgentRequest()` existed
- `src/agents/api.types.ts` — did not exist
- No `ValidationErrorCodes` enum
- No `APIResult<T>` wrapper returning typed success/failure

## Required Fix

Implement typed input validation. Every agent request must be validated against its schema before processing. Invalid requests must return a structured `ValidationErrorCodes` error with a specific code, not a generic 400 or 500.
