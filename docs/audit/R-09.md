# R-09 — Control-Plane / Repository Boundary Ambiguity

**Category:** Architecture / Control Plane
**Severity:** High

## Description

Ambiguity between ALFRED (operator/control-plane) and FortifyOS (product) repositories caused execution traces to diverge from actual code state. There was no explicit execution contract declaring the target repo/branch/commit before a task, and no enforcement that the correct repo was being inspected. This led to phantom commit claims and broken source-of-truth tracking.

## Impact

Phantom commits — claiming work was done in one repo while it existed in another. Inability to verify work by inspecting the declared repository. Broken source-of-truth model: ALFRED's records did not match the product's actual state. Operator confusion during execution about which repo was being modified. No cross-repo trace linking existed.

## Evidence

- Initial verification runs executed against `fortifyos/ALFRED` instead of `fortifyos/fortifyos`
- Execution contracts were not declared before task execution
- No `runtime/targets/` target spec existed
- Commit claims in one repo not visible when inspecting the other

## Required Fix

Explicit execution contract per task: declare target repo, branch, and expected commit before any work begins. Create and maintain `runtime/targets/fortifyos.yaml` as the source of truth for target state. Cross-repo trace linking: ALFRED manifest commits must reference the exact fortifyos commit they are based on. Strict "repo is source of truth" enforcement: never claim work exists in a repo without verifying it.

## Mapped Tasks

- Target spec creation (`runtime/targets/fortifyos.yaml`)
- Execution contract enforcement (pre/post SHA declaration per task)
- ALFRED manifest linking (commit ledger in manifest)

## Status

**RESOLVED** — Target spec exists at `fortifyos/ALFRED/runtime/targets/fortifyos.yaml`. Every task commits with an execution contract (pre-state + post-SHA). ALFRED manifest includes commit ledger linking each phase to the exact fortifyos SHA. Two-repo discipline formally established: `fortifyos/fortifyos` = product, `fortifyos/ALFRED` = operator.

## Two-Repo Discipline

| Repo | Role | Pushed to |
|------|------|-----------|
| `fortifyos/fortifyos` | Product — all product code, docs, verifiers | `origin/main` |
| `fortifyos/ALFRED` | Operator — control plane, skills, manifests | `origin/main` |

No cross-contamination: product commits go to fortifyos, operator commits go to ALFRED.
